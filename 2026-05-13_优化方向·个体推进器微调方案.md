# 优化方向：个体推进器微调（Per-Thruster Fine-Tuning）

> 编写日期：2026-05-13  
> 前置工作：v1→v2 全局模型已完成，v1 已有 `model_finetune.py` 可复用经验  
> 目标：解决"一个模型适配所有推进器导致误差较大"的问题

---

## 一、问题背景

当前 v2.0 模型使用全部 908 个正常文件（SN01-SN12）训练，服务所有推进器（SN13-SN24 测试集）。

**现有问题：**
- 全局模型 thrust RMSE = 0.0832N，但按 test_mode 分项后 ssf 高达 0.1390N
- 不同 SN 之间推力水平差可达 2 倍（SN13 mean=0.65N vs SN18 mean=0.97N），已在 v2.1 实验中发现
- 制造公差导致同一工况下不同推进器的响应特性不同

**差异化根源：**
- 推进器制造公差：催化床填充量、喷嘴尺寸、阀门响应时间差异
- 使用历史差异：cumulated_on_time / throughput / pulses 三个老化因子无法完全刻画个体差异

### 1.1 SN 编号的含义：同一型号下的不同生产个体

**SN01-SN24 是同一款单组元推进器（hydrazine monopropellant thruster）的 24 台不同生产个体，不是 24 种不同类型。** 类似于同一款手机的不同台次或同一车型的不同车辆。

所有 SN 共享相同的设计图纸和物理原理：
- 相同的催化床结构
- 相同的喷嘴口径
- 相同的阀门类型
- 相同的推进剂（肼）

但制造公差客观存在：
- 催化床填充密度差异：±3-5%
- 喷嘴加工精度差异：±1-2%
- 阀门响应时间差异：±2-3ms

v2.1 实验已经证实了这些公差的影响：**SSF 模式下 SN13 平均推力 0.65N vs SN18 平均推力 0.97N，相差约 2 倍。**

### 1.2 为什么用 SN01-12 训练、SN13-24 测试？

STFT 数据集的 24 个 SN 编号本身只是合成数据的标识符，不区分"地面试验件"和"飞行件"。我们沿用了 v1 的数据划分方式：**SN01-12 做训练集、SN13-24 做测试集**，这是标准的机器学习数据拆分。

这个拆分方式同时对应了一个合理的工程叙事：用一部分推进器的数据训练模型，去预测另一部分推进器的行为。在真实航天场景中，这种"用地面试验数据训练、用在轨飞行数据监控"的路线是标准做法。

### 1.3 微调的数据来源——在实验中的实际做法

每台推进器（SN13-SN24）在测试集中各有 **约 100 个正常文件**。微调是在实验框架下验证方案可行性的对比实验：

```
以 SN13 为例（103 个正常文件）：
  ├── 微调集（5 个文件）    ← 从 SN13 自己的文件中取出少量，在全局模型基础上继续训练
  └── 测试集（98 个文件）   ← SN13 剩下的文件，用来对比"微调前 vs 微调后"的预测精度
```

| 数据拆分 | 文件数 | 说明 |
|:---------|:------:|:------|
| 全局训练集（SN01-12） | 908 | 训练基础模型 |
| **微调集（目标 SN）** | **5 / SN** | **从该 SN 的测试文件中取 5 个，用于微调** |
| 测试集（目标 SN） | ~100 / SN | 剩余文件，评估微调效果 |

**为什么要取 5 个？** 这不是固定值——我们会做多组对比来确定最少需要多少数据：

| 微调用文件数 | 实验目的 |
|:-----------:|:---------|
| 1 | 极限少量数据下微调是否有效 |
| **5（baseline）** | **合理少量场景，主方案** |
| 10 | 数据增加后改善是否继续 |
| 50（≈一半） | 微调技术本身的上限 |

最终在研究报告中可以写："**仅用 5 条点火序列微调该推进器的专用模型，RMSE 降低 X%**"——这是用实验设计支撑的结论。至于真实航天场景中需要多少次校准点火，需要在取得 803 所的真实数据后确定。

### 1.4 全局模型 vs 微调模型的关系

| 维度 | 全局模型（v2.0） | 微调模型（v2.x） |
|:-----|:----------------|:----------------|
| 训练数据 | SN01-SN12，908 个文件 | 目标 SN 的 n 个文件（通常 10-50 个） |
| 能力 | 学会推进器的**通用物理规律** | 在通用规律基础上**适配个体偏差** |
| 精度 | thrust RMSE = 0.0832N | 预期 0.05-0.06N（-30%） |
| 置信度 | MC Dropout CV 较大（65/100） | 预期提升至 85+/100 |
| 泛化性 | 对所有 SN 有效 | 仅对目标 SN 有效 |

**微调不是否定全局模型，而是在全局模型的基础上做个性化适配。** 全局模型提供基础预测能力，微调修正个体公差导致的系统偏差。

## 三、v2 微调方案设计

### 3.1 核心思路

```
全局模型（17维 Attention-LSTM）
        ↓ 保存 .pth
个体微调（按 SN 或按 test_mode）
        ↓
多个专用模型：
  ├── dual_output_lstm_v2_sn13.pth   (SN13 专用)
  ├── dual_output_lstm_v2_sn14.pth   (SN14 专用)
  └── ...
```

### 3.2 技术路线

#### 方案 A：按 SN 微调（推荐）

每条推进器独立微调一个专用模型。

```
训练：908 文件（SN01-SN12）→ 全局模型
微调：仅使用目标 SN 的 n 个文件 → 极小 lr 微调
推理：根据输入 CSV 的 SN → 选择对应模型
```

**优势：**
- 物理意义明确（每台推进器独立）
- v1 已验证有效

**代价：**
- 测试集 12 台推进器 → 12 个专用模型 → 存储 ~43MB
- 每个模型需额外微调 ~5-10 epoch（约 10-20min 每台）

#### 方案 B：按 test_mode + SN 分组微调

对误差较高的 mode（ssf、offmod、random_short）单独微调。

**优势：**
- 针对性强（ssf 单项 RMSE=0.1390N 是主要矛盾）
- 微调数据量更充足

#### 方案 C：Factory Calibration 模式（创新方案）

每台推进器出厂时用少量点火数据做快速微调（~1 epoch，~2min），生成个体修正向量。

```
全局预测 + 个体偏置修正 = 最终预测
```

---

### 3.3 微调细节

**模型改动：** 无需改架构，只改训练脚本。

```
冻结 attention 层，仅微调 FC head + LN scale：
  - FC(256→128) 的 weight/bias
  - FC(128→2) 的 weight/bias
  - LayerNorm 的 γ/β

学习率：
  - FC head: 1e-4（主要更新层）
  - LN scale: 1e-5（微调）

数据：
  - 目标 SN 的全部正常文件
  - 用 data_pipeline_v2._build_features() 读取
  - seq_len=200, stride=100
```

**损失函数：** 与全局训练一致：`0.7×MSE(thrust) + 0.3×MSE(mfr)`

### 3.4 预期效果

| 推进器 | 全局 RMSE | 微调后预期 |
|:-------|:---------:|:----------:|
| SN13（ssf） | 0.1390 N | ~0.09 N（-35%） |
| SN13（onmod） | 0.0212 N | ~0.015 N（-29%） |
| SN13（offmod） | 0.0967 N | ~0.07 N（-28%） |
| SN18（ssf 特差） | 推测 >0.15 N | ~0.10 N |

> 参考 v1 的 27.97% 改善幅度，预期 v2 可达 20-35% 改善。

## 六、核心意义：我们不是在自导自演

### 6.1 关于合成数据的坦诚

本项目使用 Kaggle 上的 STFT 合成数据集（Spacecraft Thruster Firing Tests Dataset）。我们无法获得 803 所的真实航天数据，所有关于"地面试验/在轨飞行"的叙事均为比赛框架下的工程推演。

**但这不意味着我们在自导自演。** 以下成果不依赖数据来源，反映的是真实的工程能力。

### 6.2 我们真正交付了什么

**完整的 AI 工程闭环：**

```
数据清洗 → 特征工程（17维联合建模）
  → 模型选型（5 轮架构对比）
    → 训练 + 早停 + 可复现性验证
      → 测试（11 种 mode 分项 RMSE）
        → 可解释性（SHAP + Permutation）
          → 不确定性量化（MC Dropout）
            → 跨平台验证（CUDA → CANN 国产化）
              → 部署界面（NASA 风格大屏）
                → 个体微调方案（本文件）
```

这个 pipeline 本身是真实的。换真实数据进来，数据接口一换即可复用。

**所有技术结论经得起推敲：**

| 结论 | 实验证据 |
|:-----|:---------|
| Attention-LSTM 优于纯 LSTM | val_loss 0.000277→0.000252，且 v1 5 架构对比支持 |
| 17 维特征优于 3 维 | RMSE 0.1247→0.0832 N（-33.3%），v1 vs v2 同窗对比 -45.4% |
| 老化因子比控制信号更重要 | SHAP 显示三大老化因子占 49%，ton 仅 32% |
| dropout 在 LSTM 循环连接中不适用 | Round 1/2 实证：dropout=0.1 即破坏时序信号 |
| CANN 与 CUDA 结果一致 | S 验证中——证明算法可部署在国产算力上 |
| 负结果也有价值 | v2.1 SN Embedding 失败 → 发现训练-测试分布偏移问题 |
| 消融实验的价值 | 5 轮实验的因果链本身就是研究叙事 |

这些结论在合成数据上成立，在真实数据上大概率也成立——因为它们反映的是推进器的**物理结构和数据规律**，不是特定数据集的偶然特征。

### 6.3 比赛的评判逻辑

CRAIC 人工智能创新赛评审的是**创新思路 + 技术实现 + 落地可行性**，而非"你有没有真实航天数据"。评委清楚本科生无法取得 803 所数据。

我们展示的完整技术路线意味着：
> "我们理解问题、设计了方案、全部实现并验证了。如取得真实数据，仅需替换数据接口，pipeline 可直接复用。"

这才是基于合成数据参赛的正确姿势——不伪装数据，但用扎实的工程闭环证明技术可行性。

### 6.4 MC Dropout 与"LSTM 不能用 dropout"是否矛盾？

不矛盾。这是两个不同的事情：

| Round 1/2 的结论 | MC Dropout 的做法 |
|:-----------------|:------------------|
| **LSTM 循环连接中**加 dropout → 200 步累积衰减 → 训练崩溃 | **FC 全连接层**加 dropout=0.05，推理时开启多次前向 |
| 影响的是 LSTM 隐状态的时序传递 | 影响的是 FC 层的输出，与 LSTM 无关 |
| 结论：不要在 LSTM 层使用 dropout | 结论：在 FC 层用 dropout 做不确定性量化完全可行 |

我们的模型架构中 LSTM 层无 dropout，仅 `FC(128→2)` 前有 `Dropout(0.05)`。MC Dropout 利用这一层做多次采样，不影响 LSTM 的时序建模能力。

## 七、风险

| 风险 | 可能性 | 应对 |
|:-----|:------:|:------|
| 某些 SN 数据量不足 | 低 | 数据增强（加噪声、时间抖动） |
| 微调过拟合（仅几十个文件） | 中 | 早停 patience=5，极小 lr |
| 微调后泛化性下降 | 中 | 保留全局模型作为 fallback |
| 对 test_mode 分项改善不一致 | 中 | 按 mode+SN 交叉分组 | |
