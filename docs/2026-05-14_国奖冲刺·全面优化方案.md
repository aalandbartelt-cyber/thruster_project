## 零、评分公式深度拆解

```
总分 = ((0.8A + 0.2B) / 100) × (0.3C + 0.25D + 0.4E + 0.05F)
```

### 0.1 因子解读

**第一因子（基础分，最大100）**：
| 代码 | 维度 | 权重 | 解读 |
|:----:|:-----|:----:|:-----|
| A | 项目可行性与实用性 | **0.8** | **绝对核心**——方案能落地吗？有演示吗？有实际价值吗？ |
| B | 项目技术基础 | 0.2 | 理论基础是否扎实，技术路线是否清晰 |

**第二因子（乘数，决定上限）**：
| 代码 | 维度 | 权重 | 解读 |
|:----:|:-----|:----:|:-----|
| **E** | **团队对作品的贡献程度** | **0.4** | **乘数中最高权重**——是不是自己做的？深度如何？ |
| C | 项目创新度 | 0.3 | 技术创新点是否原创、有深度 |
| D | 项目难度 | 0.25 | 问题本身的技术难度 |
| F | 其他 | 0.05 | 附加项 |

### 0.2 冲奖公式

```
国奖 = 基础分接近满分 × 乘数尽可能大
    = (A做到极致 + B扎实) × (E证明原创深度 + C有真创新 + D体现高难度)
```

**关键洞察**：
- A（可行性实用性）权重0.8 → 不管创新多好，不能演示=基础分腰斩
- E（团队贡献）权重0.4 → 不管模型多好，看不出是自己做的=乘数打折
- C（创新度）权重0.3 → 创新点必须有实验证据支撑，不能只写PPT

### 1.4 🟡 剩余使用寿命（RUL）预测（A分 +5）

**现状**：只做异常检测（当前是否正常），不做RUL预测（还能用多久）。

**优化方案**：

| 方法 | 难度 | 说明 |
|:-----|:---:|:-----|
| **退化指标趋势外推** | 低 | 用cumulated_throughput vs thrust残差拟合退化曲线，外推至失效阈值 |
| **相似性匹配** | 中 | 找到历史上相似退化轨迹的推进器，用其寿命做参考 |
| **LSTM生存分析** | 高 | 修改输出头为Weibull参数，直接预测剩余寿命分布 |

**推荐方案**：趋势外推法（低风险、高回报）。

具体做法：
```
1. 对每台SN，按cumulated_throughput排序，计算thrust残差RMS的滑动趋势
2. 用线性/指数函数拟合退化趋势
3. 设定失效阈值（如残差RMS > 0.3N），外推至阈值对应的cumulated_throughput
4. 当前throughput → 预测失效throughput = RUL（以kg推进剂消耗量表示）
```

**视觉呈现**：在app_v2中增加"寿命仪表盘"——一个圆形进度条，绿色>70%，黄色30-70%，红色<30%，数值显示"预计剩余推进剂消耗量 XX kg"。

**答辩金句**："异常检测告诉操作员'现在有问题'，RUL预测告诉操作员'还有多少时间准备'——两者结合构成完整的预测性维护闭环。"

**预估工作量**：1天（趋势拟合+UI组件）

### 2.2 🔴 SHAP分析深化：特征交互效应（C分 +5）

| 特征对 | 交互强度 | 物理解释 |

### 2.3 🟡 因果发现：从相关到因果（C分 +4）

**现状**：SHAP是相关性分析，不等于因果。特征重要性≠特征因果性。

**优化方案**：引入**PCMCI因果发现算法**（由Jakob Runge开发，专为时间序列因果推断设计）：

```python
from tigramite import data_processing as pp
from tigramite.pcmci import PCMCI
from tigramite.independence_tests import ParCorr

# 在17维特征+thrust/mfr残差上跑PCMCI
pcmci = PCMCI(dataframe, cond_ind_test=ParCorr())
results = pcmci.run_pcmci(tau_max=5, pc_alpha=0.05)
# → 因果图（有向边）
```

**产出**：一张因果图——箭头从"因"指向"果"。对比"SHAP关联图 vs PCMCI因果图"，展示方法论升级。

**答辩金句**："我们不仅知道哪些特征与预测相关（SHAP），更进一步用因果发现算法识别了特征间的因果结构——比如cumulated_throughput是推力残差上升的因，而非相反。"

**预估工作量**：1天（安装tigramite + 数据准备 + 因果图出图）

### 2.5 🟢 对比学习异常检测（C分 +3）

**现状**：异常检测基于预测残差阈值，依赖有标签数据（270个异常标签）。

**优化方案**：用**自监督对比学习**学习正常状态的表示，异常=表示空间的距离：

```
训练：正常样本 → 数据增强(裁剪/噪声/缩放) → 正样本对 → SimCLR/TS-TCC
推理：编码→表示 → 与正常质心的距离 → 异常分数
```

**为什么加分**：这是"无监督/自监督异常检测"这一前沿方向的落地。不依赖异常标签即可检测未知异常——对应真实航天中"没见过的新故障模式"。

**不过建议**：时间紧张时可以作为"研究报告中的future work"呈现，写出技术路线+预期效果，不要求代码实现。但如果能跑出一个POC结果，就是强力加分项。

**预估工作量**：1.5天（TS-TCC实现+训练+对比实验），或0.3天（仅写future work）

## 四、D 维度优化：项目难度（权重 0.25）

### 4.1 🔴 多步超前预测（D分 +4）

**现状**：模型只预测当前时刻的thrust/mfr（一步预测）。

**优化方案**：将模型改为**多步超前预测**（multi-step-ahead）：

```python
# 当前：输入[t-T, t] → 输出[t]
# 优化：输入[t-T, t] → 输出[t+1, t+5, t+10, t+20]
self.fc = nn.Linear(256, 2 * n_horizons)  # 2输出 × 4个horizon
```

**价值**：多步预测是时序预测的前沿课题，能显著提升D分。同时有实际意义——提前预知推力衰减趋势。

**答辩金句**："我们不仅预测推进器的当前状态，还预测未来5步、10步、20步的状态——这是从'感知'到'预知'的升级。"

**预估工作量**：0.5天（修改模型输出+重新训练45分钟）

---

### 4.2 🟡 域泛化：SN-level交叉验证（D分 +3）

**现状**：train/val split是文件级随机（同一SN的文件可能同时在train和val中），这高估了泛化能力。

**优化方案**：严格的Leave-One-SN-Out交叉验证：

```
对每个SN i ∈ {01..24}：
  train: 所有 SN ≠ i 的数据
  test:  SN i 的数据
  → 记录 RMSE_i
→ 报告 24个SN的RMSE均值±标准差
```

**为什么这体现难度**：SN-level泛化比文件级泛化难得多——同一SN的train/test泄漏消失后，考验模型对"未见过推进器"的预测能力。

**预估工作量**：1天（24折训练自动化脚本 + 汇总图表）

---

### 4.3 🟡 概念漂移检测（D分 +2）

**现状**：模型训练完成后不再更新。但在真实场景中，推进器会持续老化，数据分布会漂移。

**优化方案**：实现一个轻量的**概念漂移检测器**：

```python
# 用最近N次推理的残差分布 vs 训练时的残差分布
from scipy.stats import ks_2samp
drift_detected = ks_2samp(train_residuals, recent_residuals).pvalue < 0.01
if drift_detected:
    alert("推进器特性已发生变化，建议重新标定模型")
```

**答辩金句**："我们的系统不只是'训练一次用一辈子'——内置了概念漂移检测，能在推进器特性发生变化时主动提醒重新标定。"

**预估工作量**：0.3天

